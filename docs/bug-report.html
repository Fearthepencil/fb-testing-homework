Test Case Reference TC-003: Images Load and Display Correctly

Description

During exploratory testing of the fish page for Red Snapper, one image on the page fails to load and displays a 403 Forbidden error. The error originates from an upstream server, indicating a permissions or access control issue with the image resource.

Steps to Reproduce

  1. Navigate to https://nextjs15.dev.fishingbooker.com/fish/snapper-red
  2. Log in with credentials: fishingbooker / QAFBTest (if required)
  3. Scroll through the page to view all images
  4. Observe the page for any broken or failed image loads
  5. Open browser Developer Tools (F12) and check the Network tab
  6. Look for any image requests returning 403 Forbidden status

Expected Result

All images on the fish page should load successfully and display correctly without any HTTP error codes.

Actual Result

One image fails to load and returns a 403 Forbidden error from the upstream server. The image either:

  • Displays as a broken image placeholder
  • Shows an error icon or message
  • Appears as a blank space where the image should be

The Network tab in Developer Tools shows the image request with HTTP status code 403.

Evidence

Screenshots demonstrating the issue:

Impact

  • User Experience: Degraded visual experience due to missing/broken image
  • Content Quality: Incomplete content presentation affects page professionalism
  • SEO: Broken images may negatively impact SEO if alt text is not properly configured
  • Accessibility: Users relying on images may miss important visual information

Possible Root Causes

  • Image file permissions incorrectly configured on the upstream server
  • CDN or image hosting service access control rules blocking the request
  • Expired or invalid image URL/token
  • Server-side authentication/authorization issue with the image resource
  • CORS (Cross-Origin Resource Sharing) policy blocking the image request

Suggested Fix

  1. Verify the image file exists and is accessible on the upstream server
  2. Check and correct file permissions on the upstream server (ensure read access is granted)
  3. Review CDN/image hosting service configuration and access control rules
  4. Verify the image URL is correct and not expired
  5. If using authentication tokens, ensure they are valid and properly included in the request
  6. Check CORS configuration if the image is hosted on a different domain
  7. Replace the broken image with a working alternative if the upstream issue cannot be resolved
  8. Implement proper error handling to display a fallback image or placeholder when images fail to load

Additional Notes

This issue was discovered during exploratory testing. The specific image URL and exact location on the page can be identified from the screenshots. It is recommended to test all images on the page to ensure no other images are affected by similar issues.

Summary The Filters modal on the Red Snapper fish page displays "River Fishing" as a selectable option, which is inappropriate since Red Snapper is a saltwater/marine fish species, not a freshwater river fish.
Type Functional Bug / Data Integrity Issue / Domain Logic Violation
Reported By Pavle Stefanovic
Date Reported January 2025
Status Open

Priority High
Test Environment URL: https://nextjs15.dev.fishingbooker.com/fish/snapper-red

Test Case Reference TC-004: Interactive Elements Function Correctly - Fails at "Filters apply correctly and update results" step
TC-010: Content Is Relevant to Fish Species - Violates "Charter listings are relevant to fishing for that species" requirement
Product Risk: PR-01 (Page functionality issues - HIGH), PR-03 (Inconsistent or irrelevant content - MEDIUM)

Description

Domain Logic Violation: When viewing the Red Snapper fish page and opening the Filters modal, the "Type of Fishing" section displays "River Fishing" as a selectable option. This is a fundamental domain logic error: Red Snapper (Lutjanus campechanus) is a saltwater/marine fish species that lives in ocean waters, while River Fishing implies freshwater environments. Showing Red Snapper charters under River Fishing is factually incorrect and violates core domain knowledge.

Filter Behavior Issue: When "River Fishing" filter is applied, the results show coastal Croatia sea charters with no rivers involved. This indicates that at least one of the following is broken:

  • Filter label is wrong: The filter is mislabeled and should not be called "River Fishing"
  • Filter logic is wrong: Results don't match the selected filter criteria
  • Fish-habitat mapping is wrong: Red Snapper is incorrectly tagged as available in river environments
  • Backend query ignores filter: The filter UI is displayed but the backend query doesn't respect it

This is not a preference issue or subjective content - it's a domain inconsistency that breaks search functionality. The filter options should be contextually appropriate for the fish species being viewed. For Red Snapper, appropriate fishing types would include Inshore Fishing, Nearshore Fishing, Offshore Fishing, Reef Fishing, etc., but not River Fishing or Lake Fishing (freshwater options).

Steps to Reproduce

  1. Navigate to https://nextjs15.dev.fishingbooker.com/fish/snapper-red
  2. Log in with credentials: fishingbooker / QAFBTest (if required)
  3. Wait for the page to load and charter listings to be displayed
  4. Click on the "Filters" button (located above the map on the right side)
  5. Observe the Filters modal that opens
  6. Scroll down to the "Type of Fishing" section
  7. Note that "River Fishing" is listed as an option and can be selected

Expected Result

The Filters modal should only display fishing type options that are appropriate for Red Snapper, which is a saltwater/marine fish. River Fishing and Lake Fishing (freshwater options) should not be available as filter options for this fish species. Only saltwater fishing types should be shown (e.g., Inshore Fishing, Nearshore Fishing, Offshore Fishing, Reef Fishing, Wreck Fishing, Flats Fishing, Backcountry Fishing).

Actual Result

The Filters modal displays "River Fishing" as a selectable option in the "Type of Fishing" section. This option can be checked/selected, which is inappropriate for Red Snapper since it is a saltwater fish that does not inhabit freshwater rivers. The filter system does not appear to be contextually aware of the fish species being viewed.

Evidence

Screenshot 1: Filters modal showing "River Fishing" option available and selected for Red Snapper page. The modal displays "Type of Fishing" section with "River Fishing" checked, which is inappropriate for a saltwater fish species.

Screenshot 2: Map view showing results when "River Fishing" filter is applied. The map displays coastal Croatia and Adriatic Sea locations with sea charters, demonstrating that the filter results don't match the filter criteria (no rivers are involved, only sea/coastal locations).

Impact

  • User Experience: Users looking for river fishing see sea charters, and users researching Red Snapper get misleading information. This directly affects usability and conversion rates.
  • Domain Logic Violation: Core search logic is broken - factually incorrect information is presented (saltwater fish shown with freshwater fishing option).
  • Trust & Credibility: Trust in filters and landing pages is damaged. Users may question the accuracy of all information on the site.
  • Search Functionality: Filter behavior is fundamentally incorrect - filters don't apply correctly or results don't match filter criteria.
  • SEO Trust Signals: Incorrect content and filter logic can negatively impact SEO trust signals and search engine rankings.
  • Business Impact: Poor filter accuracy leads to decreased user satisfaction, lost bookings, and damaged brand credibility.
  • Data Integrity: Fish-habitat mapping appears incorrect, suggesting broader data quality issues.

Possible Root Causes

  • Filter options are hardcoded and not dynamically filtered based on fish species characteristics
  • Missing logic to exclude freshwater fishing types for saltwater fish species
  • Fish species metadata does not include information about habitat type (saltwater vs freshwater)
  • Filter component does not receive context about the current fish species being viewed
  • Template reuse issue where the same filter options are shown for all fish species regardless of their habitat

Suggested Fix

  1. Add fish species metadata to indicate habitat type (saltwater, freshwater, or both)
  2. Implement logic to filter available fishing type options based on the fish species habitat
  3. For saltwater fish (like Red Snapper): Exclude River Fishing and Lake Fishing options
  4. For freshwater fish: Exclude saltwater-specific options (e.g., Offshore Fishing, Reef Fishing, Wreck Fishing)
  5. For fish that can be found in both: Show all options
  6. Test filter options for all fish species pages to ensure contextual appropriateness
  7. Consider adding tooltips or help text explaining why certain filters are not available for specific fish species

Additional Notes

This bug was discovered during execution of TC-004 (Interactive Elements Function Correctly) and also violates TC-010 (Content Is Relevant to Fish Species). While the filter UI functionality itself works (the modal opens, options can be selected), the domain logic and filter behavior are fundamentally broken. This is not just a content accuracy issue - it's a functional defect that breaks core search logic.

Classification: This is a Functional + Data Integrity bug with Major severity (not a crash, but core logic is wrong) and High priority (affects discoverability and trust). It maps to Product Risk PR-01 (Page functionality issues - HIGH) and PR-03 (Inconsistent or irrelevant content - MEDIUM).

Note on Geographic Context: The map shows the Adriatic Sea region (Croatia, Montenegro). Red Snapper as commonly known (Lutjanus campechanus) is native to the Gulf of Mexico and western Atlantic, not the Mediterranean/Adriatic. If the page is referring to a different snapper species native to the Mediterranean, the filter logic should still exclude freshwater options like River Fishing, as all snapper species are marine/saltwater fish.

Recommendation: Review all fish species pages to ensure filter options are appropriate for each species. Implement domain logic validation to prevent saltwater fish from showing freshwater fishing options and vice versa.

  4. Locate the <title> tag in the <head> section
  5. Observe that the title tag is present but empty

The <title> tag should be populated with a descriptive, species-specific title that is 50-60 characters long and contains the fish species name. Example: "Red Snapper Fishing Charters | FishingBooker" or "Red Snapper Fishing Trips & Charters - FishingBooker".

The <title> tag is present in the HTML but is empty (no content between opening and closing tags). This results in:

  • Browser tab showing no page title or default browser text
  • Search engines auto-generating titles from page content (uncontrolled)

  • Inconsistency with other fish pages (Tuna, Salmon) which have proper titles

Evidence

Screenshot: Page source showing empty <title> tag in the HTML head section for Red Snapper fish page.

Impact

  • SEO Ranking: Missing primary SEO ranking signal, leading to potential ranking degradation
  • Search Engine Results: Google will auto-generate title from page content, which may not be optimal for CTR
  • User Experience: Browser tab shows no descriptive title, making it difficult for users to identify the page in multiple tabs
  • Consistency: Inconsistent with other fish pages (Tuna, Salmon) which have properly populated titles
  • CTR Impact: Lower click-through rates from search results due to uncontrolled title generation
  • SEO Best Practices: Violates fundamental SEO best practices requiring unique, descriptive title tags

Possible Root Causes

  • Title tag generation logic missing or failing for Red Snapper page specifically
  • Template variable not populated for Red Snapper species
  • Data mapping issue where Red Snapper metadata is missing title field
  • Conditional rendering logic that incorrectly excludes title for this specific page

Additional Notes

This bug was discovered during execution of TC-006 (Meta Title and Description Are Present and Optimized). The empty title tag is a critical SEO failure that directly impacts search engine rankings and user experience. Test Case Reference TC-007: Header Tags (H1, H2, H3) Are Properly Structured - Fails at "Exactly one H1 tag is present" step
Product Risk: PR-02 (Poor SEO optimization affects search rankings - HIGH risk)

Description

SEO Semantics Broken: The Red Snapper fish page is missing an H1 tag, which is the primary semantic heading element that search engines use to identify the main topic of the page. The page displays "Red Snapper Fishing" text visually, but it is contained within a styled <div> element instead of a semantic H1 tag. A <div> cannot replace H1, no matter how it's styled, resulting in weakened keyword relevance, poor page topic signaling, and inconsistent SEO structure.

Accessibility Violation: Screen readers and assistive technologies rely on H1 tags as the page entry point and use heading navigation to help users understand page structure. A styled <div> is invisible to assistive technologies and breaks WCAG heading navigation, creating an accessibility defect.

Page Structure: The page has H2, H3, and H4 tags present, but no H1 tag. Visually, the page appears to have a main heading, but semantically it does not.

Steps to Reproduce

  1. Navigate to https://nextjs15.dev.fishingbooker.com/fish/snapper-red
  2. Log in with credentials: fishingbooker / QAFBTest (if required)
  3. Open page source (Right-click → View Page Source) or open DevTools → Elements tab
  4. Search for H1 tags in the HTML (Ctrl+F → search for "<h1" or "</h1>")
  5. Observe that no H1 tags are present
  6. Locate the "Red Snapper Fishing" text in the page source
  7. Note that it is contained within a <div> element with CSS classes, not an H1 tag

Expected Result

The page should have exactly one H1 tag that:

Bug Reporting Guidelines

Severity Levels (ISTQB Foundation Level v4.0.1)

Severity indicates the degree of impact of the defect on the interests of stakeholders or requirements. Severity is typically assigned by the tester who discovers the defect.

  • Critical: System crash, data loss, security vulnerability, complete feature failure, prevents core functionality
  • High: Major functionality broken, significant user impact, workaround not available, affects important features
  • Medium: Feature partially broken, moderate user impact, workaround available, affects secondary features
  • Low: Minor issue, cosmetic problem, low user impact, easy workaround, does not significantly affect functionality

Priority Levels

Priority indicates the urgency to fix the defect. Priority is typically assigned by project management or product owner based on business needs.

  • High: Must be fixed immediately, blocks other work, affects release schedule
  • Medium: Should be fixed soon, impacts user experience, should be fixed before release
  • Low: Can be fixed later, minor impact, may be deferred to future releases

Note: Severity and priority are independent. A high-severity defect may have low priority if it affects rarely-used features, while a low-severity defect may have high priority if it affects critical business processes.

Defect Status Workflow

Typical defect status values (ISTQB Foundation Level v4.0.1):

  • Open: Defect has been reported and is awaiting analysis
  • Deferred: Defect fix is postponed to a future release
  • Duplicate: Defect is a duplicate of an existing defect
  • Waiting to be fixed: Defect has been confirmed and is in the development queue
  • Awaiting confirmation testing: Defect has been fixed and is waiting for retest
  • Re-opened: Defect was closed but has reappeared or was not properly fixed
  • Closed: Defect has been fixed and confirmed, or is accepted as-is
  • Rejected: Defect is not a real defect (e.g., false positive, works as designed)

Best Practices

  • Provide clear, step-by-step reproduction instructions
  • Include screenshots or screen recordings for visual bugs
  • Test on multiple browsers/devices if applicable
  • Check if the bug is reproducible consistently
  • Document any workarounds that exist
  • Link bugs to related test cases
