Bug Report

This document contains defects discovered during testing of FishingBooker's fish page functionality. All bugs are documented following ISTQB Foundation Level v4.0.1 defect reporting standards (Section 5.5) and ISO/IEC/IEEE 29119-3 test documentation guidelines.

Defect Management Process: According to ISTQB, defect management includes a workflow for handling individual defects from discovery to closure. The workflow typically comprises activities to log, analyze, classify, decide on response, and close the defect report.

Defect Report Contents (per ISTQB v4.0.1, Section 5.5): Each defect report includes:

  • Unique identifier - Bug ID for tracking
  • Title with short summary - Brief description of the anomaly
  • Date, organization, author, role - When and by whom the defect was reported
  • Test object and test environment identification - What was being tested and in what environment
  • Context of the defect - Test case being run, test activity, SDLC phase, test technique used
  • Description of the failure - Steps to reproduce, test logs, screenshots, recordings
  • Expected results and actual results - Clear comparison
  • Severity - Degree of impact on stakeholders or requirements
  • Priority to fix - Urgency of fixing the defect
  • Status - Current state (open, deferred, duplicate, fixed, closed, rejected, etc.)
  • References - Links to related test cases, requirements, or other defects

Note: Screenshots and evidence are stored in the bug_data/ directory and referenced using relative paths.

Test Case Reference TC-003: Images Load and Display Correctly

Description

During exploratory testing of the fish page for Red Snapper, one image on the page fails to load and displays a 403 Forbidden error. The error originates from an upstream server, indicating a permissions or access control issue with the image resource.

Steps to Reproduce

  1. Navigate to https://nextjs15.dev.fishingbooker.com/fish/snapper-red
  2. Log in with credentials: fishingbooker / QAFBTest (if required)
  3. Scroll through the page to view all images
  4. Observe the page for any broken or failed image loads
  5. Open browser Developer Tools (F12) and check the Network tab
  6. Look for any image requests returning 403 Forbidden status

Expected Result

All images on the fish page should load successfully and display correctly without any HTTP error codes.

Actual Result

One image fails to load and returns a 403 Forbidden error from the upstream server. The image either:

  • Displays as a broken image placeholder
  • Shows an error icon or message
  • Appears as a blank space where the image should be

The Network tab in Developer Tools shows the image request with HTTP status code 403.

Evidence

Screenshots demonstrating the issue:

Impact

  • User Experience: Degraded visual experience due to missing/broken image
  • Content Quality: Incomplete content presentation affects page professionalism
  • SEO: Broken images may negatively impact SEO if alt text is not properly configured
  • Accessibility: Users relying on images may miss important visual information

Possible Root Causes

  • Image file permissions incorrectly configured on the upstream server
  • CDN or image hosting service access control rules blocking the request
  • Expired or invalid image URL/token
  • Server-side authentication/authorization issue with the image resource
  • CORS (Cross-Origin Resource Sharing) policy blocking the image request

Suggested Fix

  1. Verify the image file exists and is accessible on the upstream server
  2. Check and correct file permissions on the upstream server (ensure read access is granted)
  3. Review CDN/image hosting service configuration and access control rules
  4. Verify the image URL is correct and not expired
  5. If using authentication tokens, ensure they are valid and properly included in the request
  6. Check CORS configuration if the image is hosted on a different domain
  7. Replace the broken image with a working alternative if the upstream issue cannot be resolved
  8. Implement proper error handling to display a fallback image or placeholder when images fail to load

Additional Notes

This issue was discovered during exploratory testing. The specific image URL and exact location on the page can be identified from the screenshots. It is recommended to test all images on the page to ensure no other images are affected by similar issues.

BUG-002: Breadcrumb Navigation Shows Wrong Fish Species
Medium Severity Medium Priority
Bug ID BUG-002
Summary Breadcrumb navigation on the Red Snapper fish page incorrectly displays "Dolphin (Mahi Mahi)" instead of "Red Snapper", causing confusion and incorrect navigation context.
Type Functional Bug / Content Issue
Reported By Pavle Stefanovic
Date Reported January 2025
Status Open
Severity Medium - Incorrect breadcrumb affects user navigation context and content consistency, but does not prevent core functionality.
Priority Medium - Should be fixed to maintain content accuracy and user trust, but not blocking critical functionality.
Test Environment URL: https://nextjs15.dev.fishingbooker.com/fish/snapper-red
Browser: Chrome (latest version)
OS: Windows 10
Login: fishingbooker / QAFBTest
Test Case Reference TC-001: Fish Page Loads Successfully, TC-002: Navigation and Internal Links Work Correctly, TC-010: Content Is Relevant to Fish Species

Description

When viewing the Red Snapper fish page, the breadcrumb navigation displays "Home / Fish Species / Dolphin (Mahi Mahi)" instead of the expected "Home / Fish Species / Red Snapper". This is incorrect as Dolphin (Mahi Mahi) and Red Snapper are completely different fish species. The breadcrumb should accurately reflect the current page content.

Steps to Reproduce

  1. Navigate to https://nextjs15.dev.fishingbooker.com/fish/snapper-red
  2. Log in with credentials: fishingbooker / QAFBTest (if required)
  3. Observe the breadcrumb navigation located below the hero section
  4. Note the breadcrumb text displayed

Expected Result

The breadcrumb navigation should display: "Home / Fish Species / Red Snapper" (or similar, accurately reflecting the current fish species page).

Actual Result

The breadcrumb navigation displays: "Home / Fish Species / Dolphin (Mahi Mahi)", which is incorrect. The page content is about Red Snapper, but the breadcrumb shows a completely different fish species (Dolphin/Mahi Mahi).

Evidence

Impact

  • User Experience: Confusion for users who rely on breadcrumbs for navigation context
  • Content Accuracy: Inconsistent information between page content and navigation elements
  • SEO: Incorrect breadcrumb schema may negatively impact SEO and structured data
  • Trust: Users may lose trust in the accuracy of the website content
  • Navigation: Users clicking on breadcrumb links may navigate to wrong pages

Possible Root Causes

  • Hardcoded breadcrumb text that was not updated when creating the Red Snapper page
  • Template or component reuse issue where breadcrumb data is not dynamically populated based on the current page
  • Database or CMS mapping error linking the wrong fish species to the breadcrumb
  • URL routing issue where the breadcrumb is reading from incorrect page metadata
  • Copy-paste error during page creation where breadcrumb was copied from Dolphin (Mahi Mahi) page

Suggested Fix

  1. Verify the breadcrumb component is dynamically reading the correct fish species name from the page data/URL
  2. Check the database/CMS to ensure Red Snapper page has correct metadata for breadcrumb generation
  3. Update the breadcrumb to display "Home / Fish Species / Red Snapper" (or appropriate format)
  4. Verify breadcrumb links navigate to correct pages when clicked
  5. Test breadcrumb on all fish species pages to ensure this is not a systemic issue
  6. If using structured data (Schema.org BreadcrumbList), ensure it matches the visual breadcrumb
  7. Implement automated tests to catch breadcrumb mismatches in the future

Additional Notes

This bug was discovered during execution of TC-001 (Fish Page Loads Successfully). The breadcrumb is a critical navigation element that should accurately reflect the current page context. It is recommended to check all fish species pages to ensure breadcrumbs are correct across the site.

Test Case Reference TC-002: Navigation and Internal Links Work Correctly - Fails at step: "Charter listing links navigate to charter detail pages"

Description

When clicking on any charter listing card or link from the fish page (e.g., Red Snapper page), the navigation to the charter detail page fails. This prevents users from viewing detailed information about fishing charters, including pricing, availability, boat specifications, and booking options. All other navigation links in TC-002 pass successfully, but this specific step fails consistently across all charter listings.

Steps to Reproduce

  1. Navigate to https://nextjs15.dev.fishingbooker.com/fish/snapper-red
  2. Log in with credentials: fishingbooker / QAFBTest (if required)
  3. Wait for the page to load and charter listings to be displayed
  4. Locate any charter listing card in the charter listings section
  5. Click on the charter name/link or the charter card to navigate to the detail page
  6. Observe the navigation behavior

Expected Result

Clicking on a charter listing should successfully navigate to the charter detail page, displaying comprehensive information about the charter including boat details, pricing, availability calendar, reviews, photos, and booking options.

Actual Result

Clicking on any charter listing link fails to navigate to the charter detail page. The navigation does not complete successfully, preventing users from accessing charter information. The exact error or behavior observed is shown in the screenshot below.

Evidence

Impact

  • User Experience: Critical navigation failure prevents users from accessing charter details, blocking the primary user journey
  • Business Impact: Users cannot view charter information or proceed with bookings, directly impacting revenue and conversions
  • Functionality: Core feature (charter detail page access) is completely broken
  • User Trust: Broken navigation links damage user confidence in the platform
  • SEO: Internal linking issues may negatively impact SEO if search engines cannot crawl charter detail pages

Possible Root Causes

  • Broken URL routing or incorrect URL generation for charter detail pages
  • JavaScript error preventing navigation event from executing
  • Missing or incorrect href attributes on charter listing links
  • Authentication or authorization issue blocking access to charter detail pages
  • API endpoint failure when fetching charter detail data
  • Client-side routing error in Next.js application
  • 404 error for charter detail page routes that don't exist
  • Network request failure or timeout when loading charter detail page

Suggested Fix

  1. Verify charter listing links have correct href attributes pointing to valid charter detail page URLs
  2. Check browser console for JavaScript errors when clicking charter links
  3. Verify the charter detail page route exists and is properly configured in Next.js routing
  4. Test API endpoints that fetch charter detail data to ensure they return valid responses
  5. Check network tab in browser DevTools to see if requests are being made and what responses are returned
  6. Verify authentication/authorization is not blocking access to charter detail pages
  7. Test URL generation logic to ensure charter IDs/slugs are correctly formatted
  8. Check if the issue is specific to certain charter listings or affects all listings
  9. Verify client-side navigation (if using Next.js Link component) is properly configured
  10. Test on multiple browsers to rule out browser-specific issues

Additional Notes

This bug was discovered during execution of TC-002 (Navigation and Internal Links Work Correctly). All other navigation elements in TC-002 pass successfully (breadcrumb links, related species links, destination links, etc.), but charter listing links consistently fail. This is a critical issue as charter detail pages are essential for the booking flow and user decision-making process.

Bug Reporting Guidelines

Severity Levels (ISTQB Foundation Level v4.0.1)

Severity indicates the degree of impact of the defect on the interests of stakeholders or requirements. Severity is typically assigned by the tester who discovers the defect.

  • Critical: System crash, data loss, security vulnerability, complete feature failure, prevents core functionality
  • High: Major functionality broken, significant user impact, workaround not available, affects important features
  • Medium: Feature partially broken, moderate user impact, workaround available, affects secondary features
  • Low: Minor issue, cosmetic problem, low user impact, easy workaround, does not significantly affect functionality

Priority Levels

Priority indicates the urgency to fix the defect. Priority is typically assigned by project management or product owner based on business needs.

  • High: Must be fixed immediately, blocks other work, affects release schedule
  • Medium: Should be fixed soon, impacts user experience, should be fixed before release
  • Low: Can be fixed later, minor impact, may be deferred to future releases

Note: Severity and priority are independent. A high-severity defect may have low priority if it affects rarely-used features, while a low-severity defect may have high priority if it affects critical business processes.

Defect Status Workflow

Typical defect status values (ISTQB Foundation Level v4.0.1):

  • Open: Defect has been reported and is awaiting analysis
  • Deferred: Defect fix is postponed to a future release
  • Duplicate: Defect is a duplicate of an existing defect
  • Waiting to be fixed: Defect has been confirmed and is in the development queue
  • Awaiting confirmation testing: Defect has been fixed and is waiting for retest
  • Re-opened: Defect was closed but has reappeared or was not properly fixed
  • Closed: Defect has been fixed and confirmed, or is accepted as-is
  • Rejected: Defect is not a real defect (e.g., false positive, works as designed)

Best Practices

  • Provide clear, step-by-step reproduction instructions
  • Include screenshots or screen recordings for visual bugs
  • Test on multiple browsers/devices if applicable
  • Check if the bug is reproducible consistently
  • Document any workarounds that exist
  • Link bugs to related test cases
